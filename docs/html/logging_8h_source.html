218// Make a bunch of macros for logging. The way to log things is to stream
219// things to LOG(<a particular severity level>). E.g.,
220//
221// LOG(INFO) << "Found " << num_cookies << " cookies";
222//
223// You can capture log messages in a string, rather than reporting them
224// immediately:
225//
226// vector<string> errors;
227// LOG_STRING(ERROR, &errors) << "Couldn't parse cookie #" << cookie_num;
228//
229// This pushes back the new error onto 'errors'; if given a NULL pointer,
230// it reports the error via LOG(ERROR).
231//
232// You can also do conditional logging:
233//
234// LOG_IF(INFO, num_cookies > 10) << "Got lots of cookies";
235//
236// You can also do occasional logging (log every n'th occurrence of an
237// event):
238//
239// LOG_EVERY_N(INFO, 10) << "Got the " << google::COUNTER << "th cookie";
240//
241// The above will cause log messages to be output on the 1st, 11th, 21st, ...
242// times it is executed. Note that the special google::COUNTER value is used
243// to identify which repetition is happening.
244//
245// You can also do occasional conditional logging (log every n'th
246// occurrence of an event, when condition is satisfied):
247//
248// LOG_IF_EVERY_N(INFO, (size > 1024), 10) << "Got the " << google::COUNTER
249// << "th big cookie";
250//
251// You can log messages the first N times your code executes a line. E.g.
252//
253// LOG_FIRST_N(INFO, 20) << "Got the " << google::COUNTER << "th cookie";
254//
255// Outputs log messages for the first 20 times it is executed.
256//
257// Analogous SYSLOG, SYSLOG_IF, and SYSLOG_EVERY_N macros are available.
258// These log to syslog as well as to the normal logs. If you use these at
259// all, you need to be aware that syslog can drastically reduce performance,
260// especially if it is configured for remote logging! Don't use these
261// unless you fully understand this and have a concrete need to use them.
262// Even then, try to minimize your use of them.
263//
264// There are also "debug mode" logging macros like the ones above:
265//
266// DLOG(INFO) << "Found cookies";
267//
268// DLOG_IF(INFO, num_cookies > 10) << "Got lots of cookies";
269//
270// DLOG_EVERY_N(INFO, 10) << "Got the " << google::COUNTER << "th cookie";
271//
272// All "debug mode" logging is compiled away to nothing for non-debug mode
273// compiles.
274//
275// We also have
276//
277// LOG_ASSERT(assertion);
278// DLOG_ASSERT(assertion);
279//
280// which is syntactic sugar for {,D}LOG_IF(FATAL, assert fails) << assertion;
281//
282// There are "verbose level" logging macros. They look like
283//
284// VLOG(1) << "I'm printed when you run the program with --v=1 or more";
285// VLOG(2) << "I'm printed when you run the program with --v=2 or more";
286//
287// These always log at the INFO log level (when they log at all).
288// The verbose logging can also be turned on module-by-module. For instance,
289// --vmodule=mapreduce=2,file=1,gfs*=3 --v=0
290// will cause:
291// a. VLOG(2) and lower messages to be printed from mapreduce.{h,cc}
292// b. VLOG(1) and lower messages to be printed from file.{h,cc}
293// c. VLOG(3) and lower messages to be printed from files prefixed with "gfs"
294// d. VLOG(0) and lower messages to be printed from elsewhere
295//
296// The wildcarding functionality shown by (c) supports both '*' (match
297// 0 or more characters) and '?' (match any single character) wildcards.
298//
299// There's also VLOG_IS_ON(n) "verbose level" condition macro. To be used as
300//
301// if (VLOG_IS_ON(2)) {
302// // do some logging preparation and logging
303// // that can't be accomplished with just VLOG(2) << ...;
304// }
305//
306// There are also VLOG_IF, VLOG_EVERY_N and VLOG_IF_EVERY_N "verbose level"
307// condition macros for sample cases, when some extra computation and
308// preparation for logs is not needed.
309// VLOG_IF(1, (size > 1024))
310// << "I'm printed when size is more than 1024 and when you run the "
311// "program with --v=1 or more";
312// VLOG_EVERY_N(1, 10)
313// << "I'm printed every 10th occurrence, and when you run the program "
314// "with --v=1 or more. Present occurence is " << google::COUNTER;
315// VLOG_IF_EVERY_N(1, (size > 1024), 10)
316// << "I'm printed on every 10th occurence of case when size is more "
317// " than 1024, when you run the program with --v=1 or more. ";
318// "Present occurence is " << google::COUNTER;
319//
320// The supported severity levels for macros that allow you to specify one
321// are (in increasing order of severity) INFO, WARNING, ERROR, and FATAL.
322// Note that messages of a given severity are logged not only in the
323// logfile for that severity, but also in all logfiles of lower severity.
324// E.g., a message of severity FATAL will be logged to the logfiles of
325// severity FATAL, ERROR, WARNING, and INFO.
326//
327// There is also the special severity of DFATAL, which logs FATAL in
328// debug mode, ERROR in normal mode.
329//
330// Very important: logging a message at the FATAL severity level causes
331// the program to terminate (after the message is logged).
332//
333// Unless otherwise specified, logs will be written to the filename
334// "<program name>.<hostname>.<user name>.log.<severity level>.", followed
335// by the date, time, and pid (you can't prevent the date, time, and pid
336// from being in the filename).
337//
338// The logging code takes two flags:
339// --v=# set the verbose level
340// --logtostderr log all the messages to stderr instead of to logfiles
341
342// LOG LINE PREFIX FORMAT
343//
344// Log lines have this form:
345//
346// Lyyyymmdd hh:mm:ss.uuuuuu threadid file:line] msg...
347//
348// where the fields are defined as follows:
349//
350// L A single character, representing the log level
351// (eg 'I' for INFO)
352// yyyy The year
353// mm The month (zero padded; ie May is '05')
354// dd The day (zero padded)
355// hh:mm:ss.uuuuuu Time in hours, minutes and fractional seconds
356// threadid The space-padded thread ID as returned by GetTID()
357// (this matches the PID on Linux)
358// file The file name
359// line The line number
360// msg The user-supplied message
361//
362// Example:
363//
364// I1103 11:57:31.739339 24395 google.cc:2341] Command line: ./some_prog
365// I1103 11:57:31.739403 24395 google.cc:2342] Process id 24395
366//
367// NOTE: although the microseconds are useful for comparing events on
368// a single machine, clocks on different machines may not be well
369// synchronized. Hence, use caution when comparing the low bits of
370// timestamps from different machines.
